# polyglot-manual-source

A Python **or** Deno `execution: manual` processor spawns a worker (a thread in
Python, an async loop in Deno) that paces frame emission with a `MonotonicTimer`
and publishes each tick to its `frame_out` port; a Rust counting-sink plugin
subscribes and reports how many frames it received. Exercises the manual
worker-thread / async-loop idiom AND the iceoryx2 publish-from-worker path across
the polyglot boundary.

```
polyglot manual source → Rust counting sink   (frame_out → video_in)
```

## The model this example teaches

Like every streamlib app, the wiring is `add_processor` + `connect` + `start`
with **no module-loading call** and **no version at the reference site**:

```rust
let manual = runtime.add_processor(ProcessorSpec::new(
    runtime_kind.processor_ref(),   // processor_type_ref! for the python or deno manual source
    /* config */,
))?;
let sink = runtime.add_processor(ProcessorSpec::new(
    counting_sink_processor_ref(),  // processor_type_ref! for the Rust counting-sink plugin
    serde_json::Value::Null,
))?;
```

The referenced packages live in this app's **`streamlib_modules/`** folder and
the runtime lazily discovers + loads each on first reference. This app uses
three, all example-local: its own `./plugin` counting-sink Rust plugin, plus its
`./python` and `./deno` polyglot source packages.

(The `./plugin` cdylib is a member of this crate's workspace; the runner stages
its built `*.so` into `plugin/lib/` at startup before the runtime loads the
linked package.)

## Run it

```bash
./setup.sh                                  # one-time local link
cargo run -- --runtime=python
cargo run -- --runtime=deno
```

`./setup.sh` does the full local setup in one shot:

1. **SDK** — `streamlib link --engine <checkout>` points the Rust, Python, and
   Deno streamlib SDK surfaces at the in-repo checkout (crates.io patch + uv
   source + deno import-map). The SDKs aren't published yet; once they are, the
   by-version pins resolve with no link step.
2. **Packages** — `streamlib link` symlinks this example's `./plugin`, `./python`,
   and `./deno` packages into `./streamlib_modules/`.

The `streamlib` CLI must be on your `PATH` (`cargo build -p streamlib-cli`, or
`cargo install --path libs/streamlib-cli`); `setup.sh` falls back to the
checkout's built binary.

## What's committed vs generated

`streamlib_modules/`, `streamlib.lock`, `Cargo.lock`, the staged `plugin/lib/`
cdylib, and the `streamlib link --engine` override are **not committed** — they
are regenerated by `./setup.sh` (and the runner's startup staging).
