# camera-plugin-sdk-compute

Camera → an **engine-free** Rust cdylib plugin that runs a SPIR-V **compute**
kernel (grayscale) → Display. The build configuration `tatolab/drone-racer`
ships its on-GPU preprocessing under: the plugin links only the engine-free
`streamlib-plugin-sdk`, never the `streamlib` engine facade.

## The model this example teaches

The sibling `plugin/` crate is a cdylib package
(`@tatolab/camera-plugin-sdk-compute`) carrying `GrayscaleCompute`. App code
references it version-free, with no load call:

```rust
let grayscale = runtime.add_processor(ProcessorSpec::new(
    processor_type_ref!("tatolab", "camera-plugin-sdk-compute", "GrayscaleCompute"),
    serde_json::Value::Null,   // config-less processor → Null
))?;
```

`./setup.sh` links `./plugin` into `streamlib_modules/` alongside
`@tatolab/camera` and `@tatolab/display`; the runtime lazily discovers, builds,
and loads it on first reference. No manual cdylib staging, no `add_module`.

## Run it

```bash
./setup.sh    # link SDK + camera/display + ./plugin
cargo run
```

`./setup.sh` runs `streamlib link --engine <checkout>`, records
`STREAMLIB_LINK_CHECKOUT` in `.cargo/config.toml` `[env]` (so the plugin's
schema codegen resolves `@tatolab/core` from the checkout), and links the
packages. The `streamlib` CLI must be on your `PATH` (`cargo build -p streamlib-cli`).

`streamlib_modules/`, `streamlib.lock`, `Cargo.lock`, and the
`streamlib link --engine` override are gitignored (regenerated by `./setup.sh`).
