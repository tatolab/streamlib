# api-server-demo

Exercises every endpoint of the runtime's control plane ŌĆö REST (health,
registry, graph, processor/connection CRUD) plus the
`ws://127.0.0.1:<port>/ws/events` WebSocket event stream ŌĆö from an async
`#[tokio::main]` driver. Processors created through the API resolve
`SimplePassthroughProcessor` from `@tatolab/debug-utilities`.

## The model this example teaches

**The control plane is the runtime, not a loadable module.** The api-server is
statically linked into `streamlib-runtime` and served in-process, so an app
cannot instantiate it from the SDK ŌĆö you reach the control plane by running the
runtime and hitting its endpoints.

This demo is therefore self-contained: it spawns a real `streamlib-runtime`
subprocess, waits for `GET /health`, drives the control plane over reqwest +
tokio-tungstenite, then tears the subprocess down ŌĆö one `cargo run` runs the
whole thing end to end.

- There is no `add_module`, no `add_processor`, and no `streamlib` SDK
  dependency in this app. It is a pure control-plane client plus a subprocess
  supervisor.
- The spawned runtime registers the `ApiServer` in-process at boot, so the
  control plane is already up once `/health` answers.
- `@tatolab/debug-utilities` is linked into this app's **`streamlib_modules/`**
  folder so the API's dynamic-registry POST (`SimplePassthroughProcessor`) can
  resolve its provider ŌĆö the runtime discovers it there via
  `STREAMLIB_MODULES_DIR`, which the demo points at this directory.

## Run it

```bash
./setup.sh        # one-time: build the runtime binary + link debug-utilities
cargo run
```

`./setup.sh` does the full local setup in one shot:

1. **Runtime binary** ŌĆö builds `streamlib-runtime` from the checkout and records
   its path (plus `STREAMLIB_LINK_CHECKOUT`, so the runtime can build the linked
   package from source on first load) into this dir's `.cargo/config.toml [env]`
   table, so `cargo run` finds it and passes it to the spawned runtime.
2. **Packages** ŌĆö `streamlib link` symlinks `@tatolab/debug-utilities` from the
   in-repo `packages/` into `./streamlib_modules/`. Edits in that package are
   live on the next run.

The `streamlib` CLI must be on your `PATH` (build it with
`cargo build -p streamlib-cli` from the checkout, or install it with
`cargo install --path tools/streamlib-cli`). `setup.sh` falls back to the
checkout's built binary if it isn't on `PATH`.

To point at a checkout other than the repo this example ships in:

```bash
STREAMLIB_CHECKOUT=/path/to/streamlib ./setup.sh
```

## What's committed vs generated

`streamlib_modules/`, `streamlib.lock`, `.cargo/`, and `Cargo.lock` are **not
committed** (they record machine-local absolute paths and unpublished
versions). They are regenerated by `./setup.sh`, exactly like `node_modules/`.
