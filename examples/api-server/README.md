# api-server

Launches the `@tatolab/api-server` processor and waits for Ctrl+C. The runtime's
REST + WebSocket command-and-control interface serves on
`http://127.0.0.1:9000` (health, registry, graph, processor/connection CRUD,
`ws://127.0.0.1:9000/ws/events`).

## The model this example teaches

**App code never loads modules.** The whole app is `add_processor` + `start`:

```rust
runtime.add_processor(ProcessorSpec::new(
    processor_type_ref!("tatolab", "api-server", "ApiServer"),   // no version, no load call
    config,
))?;
```

- `processor_type_ref!("org", "package", "Type")` is a **version-free**
  reference. There is no `add_module`, no registry, and no version anywhere in
  `src/main.rs`.
- The referenced processor's package lives in this app's **`streamlib_modules/`**
  folder (the streamlib equivalent of `node_modules/`). On the first reference,
  the runtime discovers the providing package there and lazily loads it.

## Run it

```bash
./setup.sh        # one-time: link the SDK + the package this app uses
cargo run
```

`./setup.sh` does the full local setup in one shot:

1. **SDK** — `streamlib link --engine <checkout>` points this app's
   `streamlib = "0.6"` dependency at the in-repo SDK (a transient
   `[patch.crates-io]`). The SDK isn't published to crates.io yet; once it is,
   the bare version resolves with no link step.
2. **Packages** — `streamlib link` symlinks `@tatolab/api-server` from the
   in-repo `packages/` into `./streamlib_modules/`. Edits in that package are
   live on the next run.

The `streamlib` CLI must be on your `PATH` (build it with
`cargo build -p streamlib-cli` from the checkout, or install it with
`cargo install --path tools/streamlib-cli`). `setup.sh` falls back to the
checkout's built binary if it isn't on `PATH`.

To point at a checkout other than the repo this example ships in:

```bash
STREAMLIB_CHECKOUT=/path/to/streamlib ./setup.sh
```

## What's committed vs generated

`streamlib_modules/`, `streamlib.lock`, and `Cargo.lock` are **not committed**
(they record machine-local absolute paths and unpublished versions). They are
regenerated by `./setup.sh`, exactly like `node_modules/`.
