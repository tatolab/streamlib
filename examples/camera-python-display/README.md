# camera-python-display

Camera → a cyberpunk compositing pipeline (Python avatar + Skia overlays +
Rust blend/CRT effects + GLSL glitch) → Display. Linux-only. The richest
polyglot + surface-adapter example.

## The model this example teaches

The runner is pure glue: it references each processor with `processor_type_ref!`
(version-free) and never loads a module. Its packages live in
`streamlib_modules/`:

- `@tatolab/camera`, `@tatolab/display` — in-repo siblings.
- `@tatolab/camera-python-display-effects` — the `./effects` Rust cdylib
  (BlendingCompositor, CrtFilmGrain).
- `@tatolab/cyberpunk-processor` — the `./python` package (avatar, overlays,
  glitch subprocess processors).

The runtime lazily discovers + loads each on first reference. The host-side
surface-adapter wiring (cuda / opengl / skia pre-registration in `src/linux.rs`)
is separate application setup, not module loading.

## Run it

```bash
./setup.sh    # link SDK + camera/display + ./effects + ./python
cargo run
```

Requires `uv` (<https://docs.astral.sh/uv/>) and the `streamlib` CLI on your
`PATH` (`cargo build -p streamlib-cli`). `./setup.sh` records
`STREAMLIB_LINK_CHECKOUT` in `.cargo/config.toml` `[env]` so the effects
package's schema codegen resolves `@tatolab/core` from the checkout.

`streamlib_modules/`, `streamlib.lock`, `Cargo.lock`, and the
`streamlib link --engine` override are gitignored (regenerated by `./setup.sh`).
