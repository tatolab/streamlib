# hello-streamlib

Camera → **inline processor** → display, in one small Rust file. The
zero-ceremony proof of the milestone: an `#[processor]` authored right inside
the app, with no build step and no schema package.

## The model this example teaches

An inline processor is just a struct with a `#[processor]` attribute — no
identity string, no `streamlib.yaml`, no `schemas:`, no `build.rs`, no
`_generated_`:

```rust
#[streamlib::sdk::processor(
    execution = reactive,
    input("video_in", "@tatolab/core/VideoFrame"),
    output("video_out", "@tatolab/core/VideoFrame"),
)]
pub struct HelloForward {
    frames_forwarded: AtomicU64,
}
```

Omitting the identity string makes the macro synthesize `@app/local/HelloForward`.
`App::add_local::<HelloForward::Processor>(config)` registers it **live** — no
package on disk — and hands back a connectable handle exactly like an installed
processor:

```rust
let app = App::new()?;
let camera  = app.add(processor_type_ref!("tatolab", "camera", "Camera"), json!({}))?;
let forward = app.add_local::<HelloForward::Processor>(json!({}))?;
let display = app.add(processor_type_ref!("tatolab", "display", "Display"), json!({ /* … */ }))?;
app.connect((&camera,  "video"),     (&forward, "video_in"))?;
app.connect((&forward, "video_out"), (&display, "video"))?;
app.run()
```

The processor forwards frames verbatim (`read_raw` → `write_raw`), so it needs
no frame type and no codegen — it moves whatever bytes the camera produced
straight to the display and bumps an observable per-frame counter.

## The file list is the budget

This example is deliberately tiny:

```
Cargo.toml
setup.sh
README.md
src/main.rs           # App wiring
src/hello_forward.rs  # the inline #[processor]
```

There is no `build.rs`, no `streamlib.yaml`, no `schemas/`, and no `_generated_`.
A headless E2E in the engine workspace (`cargo test --lib -p streamlib`) keeps
that budget honest with two checks: it imports `src/hello_forward.rs` directly
and drives a fixture frame through it — so any inline dependence on a generated
or codegen module breaks the build — and it walks this directory to assert none
of those ceremony files or dirs have been reintroduced, even ones the inline
processor never references.

## Run it

```bash
./setup.sh        # one-time: link the SDK + the camera/display packages this app uses
cargo run
```

`./setup.sh` does the full local setup in one shot:

1. **SDK** — `streamlib link --engine <checkout>` points this app's
   `streamlib` dependency at the in-repo SDK (a transient
   `[patch.crates-io]`). The linked checkout is the SDK package source; there
   is no central package registry.
2. **Packages** — `streamlib link` symlinks `@tatolab/camera` and
   `@tatolab/display` from the in-repo `packages/` into `./streamlib_modules/`.
   Edits in those packages are live on the next run. The inline `HelloForward`
   processor is authored in `src/hello_forward.rs` and needs no link step.

The `streamlib` CLI must be on your `PATH` (build it with
`cargo build -p streamlib-cli` from the checkout, or install it with
`cargo install --path tools/streamlib-cli`). `setup.sh` falls back to the
checkout's built binary if it isn't on `PATH`.

To point at a checkout other than the repo this example ships in:

```bash
STREAMLIB_CHECKOUT=/path/to/streamlib ./setup.sh
```

## What's committed vs generated

`streamlib_modules/`, `streamlib.lock`, and `Cargo.lock` are **not committed**
(they record machine-local absolute paths and linked-checkout pins). They are
regenerated by `./setup.sh`, exactly like `node_modules/`.

## Live camera → display

The GPU/display run needs a real camera and window, so it is a `/verify-live`
scenario rather than a CI test. `cargo run` after `./setup.sh` is the live path;
CI exercises the frame-forwarding logic headlessly against fixture frames.
