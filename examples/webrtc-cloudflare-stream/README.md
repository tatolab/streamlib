# webrtc-cloudflare-stream

WHIP publisher streaming camera (H.264) + microphone (Opus) to Cloudflare
Stream over WebRTC. Linux-only (the H.264 encoder is Vulkan-video-backed).

## The model this example teaches

**App code never loads modules.** The whole pipeline is `add_processor` +
`connect` + `start`:

```rust
let camera = runtime.add_processor(ProcessorSpec::new(
    processor_type_ref!("tatolab", "camera", "Camera"),   // no version, no load call
    serde_json::to_value(CameraConfig::default())?,
))?;
```

- `processor_type_ref!("org", "package", "Type")` is a **version-free**
  reference. There is no `add_module`, no registry, and no version anywhere in
  `src/main.rs`.
- Each referenced processor's package lives in this app's **`streamlib_modules/`**
  folder (the streamlib equivalent of `node_modules/`). On the first reference,
  the runtime discovers the providing package there and lazily loads it.
- The typed config structs (`CameraConfig`, `H264EncoderConfig`, ŌĆ”) are
  generated at **build time** by `streamlib-jtd-codegen` (see `build.rs`) from
  the schemas declared in `streamlib.yaml`. That codegen resolves its schema
  deps from the checkout via `STREAMLIB_LINK_CHECKOUT`, which `./setup.sh` wires
  up ŌĆö so the committed `streamlib.yaml` is load-bearing here and must stay.

## Run it

```bash
./setup.sh        # one-time: link the SDK + the packages this app uses
cargo run
```

`./setup.sh` does the full local setup in one shot:

1. **SDK** ŌĆö `streamlib link --engine <checkout>` points this app's
   `streamlib = "0.6"` dependency (and the `streamlib-jtd-codegen` build-dep) at
   the in-repo SDK (a transient `[patch.crates-io]`), and publishes
   `STREAMLIB_LINK_CHECKOUT` to this dir's cargo `[env]` so `build.rs` codegen
   resolves schema deps from the checkout. The SDK isn't published to crates.io
   yet; once it is, the bare version resolves with no link step.
2. **Packages** ŌĆö `streamlib link` symlinks `@tatolab/audio`, `@tatolab/camera`,
   `@tatolab/h264`, `@tatolab/opus`, and `@tatolab/webrtc` from the in-repo
   `packages/` into `./streamlib_modules/`. Edits in those packages are live on
   the next run.

The `streamlib` CLI must be on your `PATH` (build it with
`cargo build -p streamlib-cli` from the checkout, or install it with
`cargo install --path tools/streamlib-cli`). `setup.sh` falls back to the
checkout's built binary if it isn't on `PATH`.

To point at a checkout other than the repo this example ships in:

```bash
STREAMLIB_CHECKOUT=/path/to/streamlib ./setup.sh
```

## What's committed vs generated

`streamlib_modules/`, `streamlib.lock`, and `Cargo.lock` are **not committed**
(they record machine-local absolute paths and unpublished versions). They are
regenerated by `./setup.sh`, exactly like `node_modules/`.

`streamlib.yaml` (declaring the schema deps the `build.rs` codegen generates
typed config structs from) **is** committed and load-bearing for the build.
